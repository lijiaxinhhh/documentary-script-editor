# MVP 评估计划

## 人工检查

1. 打开本地站点。
2. 不看 README，确认首屏能理解工具价值和主要下一步。
3. 点击“载入样例”。
4. 点击逐字稿里的“播放”“高亮”“入稿”。
5. 在故事版选择一个模板，并移动一个视频卡片。
6. 在故事版添加图片资料卡和注释卡，确认卡片不撑破栏目。
7. 导出“剪辑软件导入说明”，确认文件能下载。
8. 打开“提交需求”反馈面板。
9. 空提交应出现错误提示。
10. 填写表单后应显示本机保存成功。
11. 检查移动端 390px 宽度无横向滚动。

## Playwright E2E

覆盖路径：

- 首屏出现价值表达、导入视频、载入样例、提交需求。
- 载入样例后出现逐字稿和故事版。
- 入稿后故事版出现视频卡片。
- 故事版可以添加图片资料卡，图片文件名可见。
- 反馈表单空提交显示错误，填写后成功。
- 反馈保存后 JSON/CSV 导出按钮可用。
- 本地项目 JSON 导出能触发下载。
- 剪辑软件导入说明能触发下载。
- 导入无视频的本地项目后，页面提示重新关联视频；选择视频后恢复到文字剪辑就绪状态。
- 关联视频文件名不一致时显示提示。
- 载入样例后自动保存最近项目，刷新后可用“最近项目”恢复。
- 导入视频后可创建转写任务；本机执行层未启动时显示“等待执行层”和“重新检查”。
- 本机桥接服务 `/health` 能返回 runtime 探测结果，`/transcribe` 在缺少 ASR runtime 时返回受控错误。
- OpenAI 转写路径会把用户输入的 Key 作为 `X-API-Key` 传给本机桥，并把桥接返回写入逐字稿。
- 通义/百炼小文件转写路径会把用户输入的 Key 作为 `X-API-Key` 传给本机桥，并在缺 Key 时返回受控错误。
- Deepgram 转写路径会把用户输入的 Key 作为 `X-API-Key` 传给本机桥，并在缺 Key 时返回受控错误。
- 桥接返回 speaker / words 时，前端能保存发言人和词级时间戳。
- 390x844 移动端无横向溢出。

## Generator / Evaluator 循环

参考 Anthropic 工程博客的 harness 思路，把本轮工作拆成三步：

1. Generator：根据 JTBD 和核心路径改造页面。
2. Evaluator：用 UX rubric、Playwright、axe 和构建命令检查结果。
3. Repair：最多 3 轮修复明确失败项，不把“看起来完成”当作完成。

参考链接：https://www.anthropic.com/engineering/harness-design-long-running-apps

## 单元测试

覆盖对象：

- 剪辑软件导出函数在有高亮时生成时间线内容。
- SRT 导出包含发言人和正确字幕顺序。
- 本地项目导出不会保存浏览器临时视频地址。
- 本地项目恢复能重新关联同一会话里的视频对象。
- 反馈导出 JSON/CSV 包含完整字段。

## Accessibility 检查

使用 `@axe-core/playwright` 做基础扫描，重点关注：

- 表单字段 label。
- 按钮可识别名称。
- 明显颜色对比问题。
- 键盘焦点和可交互元素语义。

## Lighthouse / 性能

当前 MVP 优先使用构建体积和手动浏览器检查。Lighthouse 可作为下一轮加入，重点检查：

- 首屏加载。
- 可访问性。
- Best Practices。

## 失败处理策略

测试最多修复 3 轮。若仍失败，在 `docs/evaluation-report.md` 中记录失败原因、复现步骤和下一轮建议。
